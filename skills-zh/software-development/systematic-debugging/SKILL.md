# 系统化调试

## 概述

随机的修复浪费时间和制造新 bug。快速补丁掩盖根本问题。

**核心原则：** 在尝试修复之前，总是先找到根因。症状修复是失败。

**违背这个流程的字面意思，就是违背调试的精神。**

## 铁律

```
不进行根因调查，就不能修复
```

如果你没完成第一阶段，你就不能提出修复方案。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别要使用，当：**
- 时间紧迫时（紧急情况让人想猜测）
- "就快速修一下"看起来很明显时
- 已经尝试过多次修复时
- 之前的修复没有生效时
- 没有完全理解问题时

**不要跳过，当：**
- 问题看起来很简单（简单 bug 也有根因）
- 很着急时（匆忙保证返工）
- 有人要你立刻修好时（系统化比乱撞更快）

## 四阶段

你必须完成每个阶段后才能进入下一阶段。

---

## 阶段一：根因调查

**在尝试任何修复之前：**

### 1. 仔细阅读错误信息

- 不要跳过错误或警告
- 它们通常包含精确的解决方案
- 完整阅读堆栈跟踪
- 记录行号、文件路径、错误代码

**行动：** 用 `read_file` 读取相关源文件。用 `search_files` 在代码库中搜索错误字符串。

### 2. 一致地重现

- 能可靠地触发它吗？
- 精确步骤是什么？
- 每次都发生吗？
- 如果不能重现 → 收集更多数据，不要猜测

**行动：** 用 `terminal` 工具运行失败的测试或触发 bug：

```bash
# 运行特定的失败测试
pytest tests/test_module.py::test_name -v

# 用详细输出运行
pytest tests/test_module.py -v --tb=long
```

### 3. 检查最近的改动

- 什么改动可能导致这个问题？
- Git diff、最近的 commit
- 新依赖、配置改动

**行动：**

```bash
# 最近的 commits
git log --oneline -10

# 未提交的改动
git diff

# 特定文件的改动
git log -p --follow src/problematic_file.py | head -100
```

### 4. 在多组件系统中收集证据

**当系统有多个组件时（API → 服务 → 数据库，CI → 构建 → 部署）：**

**在提出修复之前，添加诊断工具：**

对于每个组件边界：
- 记录进入组件的数据
- 记录离开组件的数据
- 验证环境/配置传播
- 检查每层的状态

运行一次收集证据，显示在哪里坏了。
然后分析证据找出故障组件。
然后调查那个特定组件。

### 5. 追踪数据流

**当错误在调用栈深处时：**

- 坏值从哪里来的？
- 什么调用这个函数传入了坏值？
- 一直向上追踪直到找到源头
- 在源头修复，不是在症状处修复

**行动：** 用 `search_files` 追踪引用：

```python
# 找到函数被调用的地方
search_files("function_name(", path="src/", file_glob="*.py")

# 找到变量被设置的地方
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
```

### 阶段一完成清单

- [ ] 错误信息完整阅读并理解
- [ ] 问题一致地重现
- [ ] 最近的改动已识别并审查
- [ ] 证据已收集（日志、状态、数据流）
- [ ] 问题隔离到特定组件/代码
- [ ] 形成了根因假设

**停：** 在理解 WHY 发生之前，不要进入阶段二。

---

## 阶段二：模式分析

**修复之前先找模式：**

### 1. 找工作的例子

- 在同一代码库中定位类似的工作代码
- 什么工作了，与什么坏了的类似？

**行动：** 用 `search_files` 找类似模式：

```python
search_files("similar_pattern", path="src/", file_glob="*.py")
```

### 2. 与参考对比

- 如果要实现一个模式，完整阅读参考实现
- 不要略读——每一行都读
- 在应用之前完全理解这个模式

### 3. 识别差异

- 工作和坏了之间有什么不同？
- 列出每个差异，无论多小
- 不要假设"那个不重要"

### 4. 理解依赖

- 这个需要什么其他组件？
- 什么设置、配置、环境？
- 它做什么假设？

---

## 阶段三：假设和测试

**科学方法：**

### 1. 形成一个假设

- 清楚地陈述："我认为 X 是根因因为 Y"
- 写下来
- 要具体，不要模糊

### 2. 最少测试

- 做最小可能的改动来测试假设
- 一次一个变量
- 不要同时修多件事

### 3. 验证后再继续

- 有效？→ 阶段四
- 无效？→ 形成新的假设
- 不要在顶上加更多修复

### 4. 当不知道时

- 说"我不理解 X"
- 不要假装知道
- 问用户帮忙
- 做更多研究

---

## 阶段四：实施

**修复根因，不是症状：**

### 1. 创建失败的测试用例

- 最简单的重现
- 如果可以，自动化测试
- 修复前必须有
- 使用 `test-driven-development` 技能

### 2. 实施单一修复

- 解决已识别的根因
- 一次只改一个
- 不要"顺手"改进
- 不要捆绑重构

### 3. 验证修复

```bash
# 运行特定的回归测试
pytest tests/test_module.py::test_regression -v

# 运行完整套件 — 无回归
pytest tests/ -q
```

### 4. 如果修复不生效 — 三次规则

- **停。**
- 计数：你尝试了多少次修复？
- 如果 < 3：回到阶段一，用新信息重新分析
- **如果 ≥ 3：停并质疑架构（见下面第 5 步）**
- 不要在没有架构讨论的情况下尝试第四次修复

### 5. 如果 3+ 次修复失败：质疑架构

**表明架构问题的模式：**
- 每个修复在不同地方揭示新的共享状态/耦合
- 修复需要"大规模重构"才能实现
- 每个修复在其他地方产生新症状

**停并质疑基础：**
- 这个模式根本上是健全的吗？
- 我们是在"靠惯性坚持"吗？
- 相比继续修复症状，我们应该重构架构吗？

**在尝试更多修复之前与用户讨论。**

这不是假设失败——是错误的架构。

---

## 红旗 — 停，遵循流程

如果发现自己这样想：
- "先快速修复，以后调查"
- "就试试改 X 看看是否有效"
- "加多个改动，运行测试"
- "跳过测试，我手动验证"
- "可能是 X，我修那个"
- "我不太理解但这可能有效"
- "模式说 X 但我要不同地适应"
- "主要问题是：[列出修复但不调查]"
- 在追踪数据流之前提出解决方案
- **"再试一次修复"（当已经尝试了 2+ 次时）**
- **每个修复在不同地方揭示新问题**

**以上全部意味着：停。回到阶段一。**

**如果 3+ 次修复失败：** 质疑架构（阶段四第 5 步）。

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要流程" | 简单问题也有根因。流程对简单 bug 也快。 |
| "紧急，没时间走流程" | 系统化调试比猜和乱撞更快。 |
| "先试试这个，然后调查" | 第一次修复设定了模式。从一开始就用正确的方式做。 |
| "修复生效后再写测试" | 未测试的修复不会持久。测试优先才能证明。 |
| "一次修多个节省时间" | 无法隔离什么有效了。会造成新 bug。 |
| "参考太长，我会适应模式" | 部分理解保证 bug。完整阅读它。 |
| "我看到问题了，让我修" | 看到症状 ≠ 理解根因。 |
| "再试一次修复"（2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要再次修复。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|---------|---------|
| **1. 根因** | 阅读错误、重现、检查改动、收集证据、追踪数据流 | 理解 WHAT 和 WHY |
| **2. 模式** | 找工作例子、对比、识别差异 | 知道哪里不同 |
| **3. 假设** | 形成理论、最少测试、一次一个变量 | 已确认或新假设 |
| **4. 实施** | 创建回归测试、修复根因、验证 | Bug 已解决，所有测试通过 |

## Hermes Agent 集成

### 调查工具

在阶段一使用这些 Hermes 工具：

- **`search_files`** — 找错误字符串、追踪函数调用、定位模式
- **`read_file`** — 用行号读取源代码以便精确分析
- **`terminal`** — 运行测试、检查 git 历史、重现 bug
- **`web_search`/`web_extract`** — 研究错误信息、库文档

### 结合 delegate_task

对于复杂多组件调试，派发调查子 Agent：

```python
delegate_task(
    goal="调查为什么 [特定测试/行为] 失败",
    context="""
    遵循 systematic-debugging 技能：
    1. 仔细阅读错误信息
    2. 重现问题
    3. 追踪数据流找到根因
    4. 报告发现 — 还不要修复

    错误：[粘贴完整错误]
    文件：[失败代码的路径]
    测试命令：[精确命令]
    """,
    toolsets=['terminal', 'file']
)
```

### 结合 test-driven-development

修复 bug 时：
1. 写重现 bug 的测试（RED）
2. 系统化调试找到根因
3. 修复根因（GREEN）
4. 测试证明修复并防止回归

## 现实影响

从调试会话中：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时乱撞
- 首次修复率：95% vs 40%
- 引入新 bug：接近零 vs 常见

**没有捷径。没有猜测。系统化永远胜出。**
