# 测试驱动开发（TDD）

## 概述

先写测试。看它失败。写最少的代码让它通过。

**核心原则：** 如果你没看过测试失败，你不知道它测的是不是对的东西。

**违背规则的字面意思，就是违背规则的精神。**

## 何时使用

**始终使用：**
- 新功能
- Bug 修复
- 重构
- 行为变更

**例外（先问用户）：**
- 一次性原型
- 生成的代码
- 配置文件

想"就这一次跳过 TDD"？停手。这是自我合理化。

## 铁律

```
没有失败的测试，就不能写生产代码
```

在测试之前就写了代码？删掉。重写。

**没有例外：**
- 不要保留作为"参考"
- 不要在写测试时"调整"它
- 不要去看它
- 删掉就是删掉

从头根据测试重新实现。就这样。

## 红-绿-重构循环

### RED — 写失败的测试

写一个最简测试，展示应该发生什么。

**好测试：**
```python
def test_retries_failed_operations_3_times():
    attempts = 0
    def operation():
        nonlocal attempts
        attempts += 1
        if attempts < 3:
            raise Exception('fail')
        return 'success'

    result = retry_operation(operation)

    assert result == 'success'
    assert attempts == 3
```
名称清晰，测试真实行为，一次一件事。

**坏测试：**
```python
def test_retry_works():
    mock = MagicMock()
    mock.side_effect = [Exception(), Exception(), 'success']
    result = retry_operation(mock)
    assert result == 'success'  # 重试次数呢？时机呢？
```
名称模糊，测试 mock 而不是真实代码。

**要求：**
- 每个测试一个行为
- 清晰的描述性名称（名称里有"and"？拆开）
- 真实代码，不是 mocks（除非真的无法避免）
- 名称描述行为，不是实现

### 验证 RED — 看它失败

**强制要求。永远不要跳过。**

```bash
# 用 terminal 工具运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v
```

确认：
- 测试失败（不是拼写错误导致的错误）
- 失败信息符合预期
- 失败是因为功能缺失

**测试立即通过？** 你在测试已有行为。修复测试。

**测试报错？** 修复错误，重新运行直到它正确失败。

### GREEN — 最少代码

写通过测试的最简单代码。不要多加。

**好：**
```python
def add(a, b):
    return a + b  # 没有额外的东西
```

**坏：**
```python
def add(a, b):
    result = a + b
    logging.info(f"Adding {a} + {b} = {result}")  # 多余的！
    return result
```

不要添加功能、不要重构其他代码、不要在通过测试之外做任何"改进"。

**GREEN 阶段可以作弊：** 可以硬编码返回值、复制粘贴、重复代码、跳过边界情况。这些在 REFACTOR 阶段再修。

### 验证 GREEN — 看它通过

**强制要求。**

```bash
# 运行特定测试
pytest tests/test_feature.py::test_specific_behavior -v

# 然后运行全部测试检查回归
pytest tests/ -q
```

确认：
- 测试通过
- 其他测试仍然通过
- 输出干净（无错误、无警告）

**测试失败？** 修代码，不是测试。

**其他测试失败？** 现在修回归。

### REFACTOR — 清理

只在 green 之后：
- 消除重复
- 改进命名
- 提取辅助函数
- 简化表达式

保持测试全程 green。不要添加行为。

**如果在重构过程中测试失败：** 立即撤销。步子迈小一点。

### 重复

下一个失败测试对应下一个行为。一次一个循环。

## 为什么顺序重要

**"我写完代码再测来验证它能用"**

先写的测试立即通过。立即通过什么都证明不了：
- 可能测的是错的东西
- 可能测的是实现，不是行为
- 可能漏掉你忘记的边界情况
- 你从来没见它抓住过 bug

测试优先强迫你看到测试失败，证明它真的测了什么东西。

**"我已经手动测了所有边界情况"**

手动测试是随意的。你以为测了一切，但：
- 没有记录你测了什么
- 代码变动时无法重跑
- 压力下容易忘记情况
- "我试的时候能用" ≠ 全面

自动化测试是系统化的。每次以相同方式运行。

**"删掉 X 小时的工作是浪费"**

沉没成本谬误。时间已经过去了。你现在的选择：
- 删除并用 TDD 重写（高置信度）
- 保留并事后补测试（低置信度，可能有 bug）

"浪费"是保留你无法信任的代码。

**"TDD 是教条，务实意味着适应"**

TDD 是务实的：
- 在提交前发现 bug（比事后调试快）
- 防止回归（测试立即抓住破坏）
- 记录行为（测试展示如何使用代码）
- 支持重构（自由改动，测试抓住破坏）

"务实"的捷径 = 在生产环境调试 = 更慢。

**"事后测试同样能达到目标——重要的是精神不是仪式"**

不对。事后测试回答"这干什么用？"测试优先回答"这应该干什么用？"

事后测试被实现偏见污染。你测的是你构建的，不是需要的。测试优先在实现前就强迫发现边界情况。

## 常见自我合理化

| 借口 | 现实 |
|------|------|
| "太简单不需要测" | 简单代码也会坏。测试只需 30 秒。 |
| "我事后测" | 事后测试立即通过，什么都证明不了。 |
| "事后测试同样能达到目标" | 事后 = "这干什么用？" 测试优先 = "这应该干什么用？" |
| "已经手动测过了" | 随意 ≠ 系统化。无记录，无法重跑。 |
| "删掉 X 小时是浪费" | 沉没成本谬误。保留未验证代码是技术债务。 |
| "保留作为参考，先写测试" | 你会调整它。这等于事后测试。删掉就是删掉。 |
| "需要先探索" | 可以。探索结果扔掉，从 TDD 开始。 |
| "测试难 = 设计不清楚" | 听测试的。难测 = 难用。 |
| "TDD 会拖慢我" | TDD 比调试快。务实 = 测试优先。 |
| "手动测试更快" | 手动不能证明边界情况。每次改动你都要重测。 |
| "现有代码没有测试" | 你在改进它。为你改动的代码添加测试。 |

## 红旗 — 停，从头来

如果发现自己做了这些，删掉代码，用 TDD 重来：

- 先写代码后写测试
- 事后补测试
- 第一次运行测试就通过
- 解释不清为什么测试失败
- "以后"再补测试
- 自我合理化"就这一次"
- "我已经手动测过了"
- "事后测试能达到同样目的"
- "保留作为参考"或"调整现有代码"
- "已经花了 X 小时，删掉浪费"
- "TDD 是教条，我很务实"
- "这个不一样因为……"

以上全部意味着：删代码。用 TDD 重来。

## 验证清单

标记工作完成前：

- [ ] 每个新函数/方法都有测试
- [ ] 每个测试在实现前都看过它失败
- [ ] 每个测试失败原因符合预期（功能缺失，不是拼写错误）
- [ ] 为通过每个测试写了最简代码
- [ ] 所有测试通过
- [ ] 输出干净（无错误、无警告）
- [ ] 测试使用真实代码（只在无法避免时用 mocks）
- [ ] 覆盖了边界情况和错误

有项打不了勾？你跳过了 TDD。从头来。

## 卡住时

| 问题 | 解决方案 |
|------|---------|
| 不知道怎么测 | 写你想要调用的 API。先写断言。问用户。 |
| 测试太复杂 | 设计太复杂。简化接口。 |
| 什么都得 mock | 代码太耦合。用依赖注入。 |
| 测试设置巨大 | 提取辅助函数。还复杂？简化设计。 |

## Hermes Agent 集成

### 运行测试

每一步用 `terminal` 工具运行测试：

```python
# RED — 验证失败
terminal("pytest tests/test_feature.py::test_name -v")

# GREEN — 验证通过
terminal("pytest tests/test_feature.py::test_name -v")

# 完整套件 — 验证无回归
terminal("pytest tests/ -q")
```

### 结合 delegate_task

派发子 Agent 实现时，在 goal 里强制 TDD：

```python
delegate_task(
    goal="用严格 TDD 实现 [功能]",
    context="""
    遵循 test-driven-development 技能：
    1. 先写失败的测试
    2. 运行测试验证它失败
    3. 写最简代码通过测试
    4. 运行测试验证它通过
    5. 如需要则重构
    6. 提交

    项目测试命令：pytest tests/ -q
    项目结构：[描述相关文件]
    """,
    toolsets=['terminal', 'file']
)
```

### 结合 systematic-debugging

发现 bug？写重现它的失败测试。遵循 TDD 循环。测试证明修复并防止回归。

永远不要不带测试修 bug。

## 测试反模式

- **测试 mock 行为而不是真实行为** — mocks 应该验证交互，而不是替代被测系统
- **测试实现细节** — 测试行为/结果，不是内部方法调用
- **只有愉快路径** — 始终测试边界情况、错误和限制
- **脆弱测试** — 测试应验证行为，不是结构；重构不应打破它们

## 最终规则

```
生产代码 → 测试存在且先失败
否则 → 不是 TDD
```

没有用户明确许可，不得例外。
